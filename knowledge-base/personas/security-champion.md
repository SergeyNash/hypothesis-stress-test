# Persona: Security Champion

## Role Summary

A Security Champion is a software engineer who takes additional responsibility for security within a development team.

The role acts as a bridge between AppSec and developers, helping translate AppSec requirements into practical engineering work and communicating team constraints back to security stakeholders.

## Goals

- Embed security into daily development workflows.
- Reduce vulnerabilities reaching production.
- Reduce frustration caused by security requirements by explaining their purpose.
- Help the team adopt secure practices.
- Represent both AppSec reasoning and developer constraints.

## Responsibilities

- Raise security concerns during planning and backlog discussions.
- Perform initial triage of findings from SAST, DAST, and other tools.
- Help the team adopt secure practices such as secret management, authentication handling, and input validation.
- Explain business context and technical limitations to AppSec.
- Organize security knowledge-sharing sessions.
- Ensure security considerations are included in code reviews.

## Recurring Pains

- Too many false positives from SAST tools.
- Being positioned as the "bad cop" inside the development team.
- Time constraints from balancing feature work and security responsibilities.
- Limited deep security expertise compared to AppSec specialists.
- Difficulty balancing team velocity and security quality.
- Security tools that do not integrate smoothly into DevOps processes.

## Decision Criteria

- Can this fit into the team's normal development workflow?
- Does it reduce security friction without lowering quality?
- Can it help the team prioritize vulnerabilities pragmatically?
- Is it clear enough for non-specialist developers to use?
- Does it strengthen collaboration with AppSec?

## Success Metrics

- Fewer critical vulnerabilities reaching production.
- Lower average remediation time.
- Team proactively raises security concerns.
- Penetration testing results improve over time.
- AppSec stakeholders are satisfied with collaboration quality.

## Objections / Anti-Patterns

- Acting as a "mini-AppSec" and handling everything alone.
- Sacrificing delivery entirely for security, or ignoring security for delivery.
- Passing security requirements to the team without business or technical context.
- Turning security into punishment instead of shared responsibility.

## Buying / Influence Role

Internal adoption bridge and workflow influencer. Usually not a buyer, but important for team-level adoption, developer sentiment, feedback loops, and practical rollout.
